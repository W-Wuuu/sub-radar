# 好商业模式评估框架 V1.1

> 创建日期：2026-06-21
> 更新日期：2026-06-21（V1.1：用户审批通过，追加第四红线、虚拟MVP、消费者硬指标、禁止清单）
> 状态：✅ 已审批，执行中
> 使用方式：对每个候选赛道逐项打分（1-5分），加权总分 ≥ 3.5 进入候选池。

---

## 一、框架设计哲学

评估一个商业模式是否"好"，不是看好不好看、热不热门，而是看它在**低人工、高自动化、可持续**三个维度上，能否稳定产生现金流。

本框架采用**五维十四项**评估体系 + **四条红线** + **禁止清单**。

---

## 二、红线检查（一票否决，先于打分）

| 红线 # | 内容 | 逻辑 |
|--------|------|------|
| 🔴 R1 | 痛点子项（1.1/1.2/1.3）任一 ≤ 2分 | 痛点不够真，不值得做 |
| 🔴 R2 | 自动化率 < 80%（即 2.1 ≤ 3分） | 无法自动化，不符合项目定位 |
| 🔴 R3 | 毛利率 < 60%（即 3.1 ≤ 2分） | 盈利质量不达标 |
| 🔴 R4 | 无法清晰回答"用户为什么下个月还会用/续费" | 没有复购逻辑 = 不可持续 |

**R4 补充说明**：必须用一句话说清续费驱动力。例如：
- ✅ "用户的历史数据累积在平台，离开即丢失分析记录"
- ✅ "问题每月自然重复发生（如月报、账单、健康复查）"
- ❌ "用户喜欢产品"（不是续费逻辑，是情感描述）

## 三、五维评估体系

### 维度1：痛点真实性（权重 25%）

| # | 评估项 | 评分标准（1-5） | 关键问题 |
|---|--------|---------------|---------|
| 1.1 | 痛点频率 | 5=每日发生，1=一年一次 | 用户多久遇到一次这个问题？ |
| 1.2 | 痛点强度 | 5=不解决就无法正常生活/工作，1=锦上添花 | 这个问题让用户多痛苦？ |
| 1.3 | 付费意愿 | 5=已有大量付费替代方案且用户习惯付费，1=从未有人为此付费 | 用户是否已证明过愿意为此付费？ |

**R1**：任一子项 ≤ 2 分 → 直接淘汰。

---

### 维度2：自动化可行性（权重 30%）

| # | 评估项 | 评分标准（1-5） | 关键问题 |
|---|--------|---------------|---------|
| 2.1 | 交付自动化率 | 5=100%可自动化（无需人工），1=核心环节必须人工 | 从获客到交付到售后，能否全链路自动化？ |
| 2.2 | 技术成熟度 | 5=现有AI/API可直接完成，1=需要前沿研究突破 | 当前技术水平能否支撑自动化？ |
| 2.3 | 人工断点数量 | 5=零人工断点，1=≥5个人工断点 | 哪些环节需要我（或人工）介入？ |

**R2**：自动化率 < 80%（得分 ≤ 3）→ 直接淘汰。

---

### 维度3：盈利质量（权重 20%）

| # | 评估项 | 评分标准（1-5） | 关键问题 |
|---|--------|---------------|---------|
| 3.1 | 毛利率 | 5=>90%，4=80-90%，3=60-80%，2=40-60%，1=<40% | 每单的边际成本是多少？ |
| 3.2 | 复购/订阅逻辑 | 5=自然月付/年付订阅，1=纯一次性交易 | 收入是否可预测、可累积？ |
| 3.3 | 定价空间 | 5=可定价$50+/月且用户不敏感，1=< $5/月 | 是否有足够的利润空间支撑增长？ |

**R3**：毛利率 < 60% → 直接淘汰。

---

### 维度4：规模化潜力（权重 15%）

| # | 评估项 | 评分标准（1-5） | 关键问题 |
|---|--------|---------------|---------|
| 4.1 | TAM（可寻址市场） | 5=全球亿级用户，1=万人级小众 | 市场天花板在哪？ |
| 4.2 | 获客可复制性 | 5=SEO/ASO/PLG可规模化获客，1=必须销售团队或关系驱动 | 能否不依赖人脉和销售获客？ |

---

### 维度5：防御性与风险（权重 10%）

| # | 评估项 | 评分标准（1-5） | 关键问题 |
|---|--------|---------------|---------|
| 5.1 | 护城河 | 5=数据网络效应+品牌+切换成本，1=可被一周复制 | 建立后能否防御竞争？ |
| 5.2 | 风险可控性 | 5=零合规/道德/平台风险，1=高风险 | 是否容易被平台封杀或政策打击？ |

---

## 四、禁止清单（自动淘汰，不进入评估）

以下方向不探索、不评估、不投入：

| ❌ # | 禁止项 | 原因 |
|------|--------|------|
| B1 | 需要人工客服介入的服务 | 违背"零人工执行"约束 |
| B2 | 内容平台流量套利（批量做号、搬运、洗稿） | 不可持续，平台风险极高 |
| B3 | 金融/医疗/法律合规高风险领域 | 除非合规成本为0 |
| B4 | 依赖人脉/关系/线下资源的模式 | 不可复制，不可自动化 |
| B5 | 纯流量套利、擦边内容、低质量信息差收割 | 用户明确排斥 |

## 五、虚拟MVP验证流程（资源投入前必须完成）

在正式投入任何资源（域名/API/服务器）之前，必须先通过以下验证：

```
┌──────────────────────┐
│ 1. 问题访谈（5-10人） │
│ 目标用户是否承认痛点？ │
└────────┬─────────────┘
         │ 验证通过
┌────────▼─────────────┐
│ 2. 方案描述测试       │
│ 展示方案，他们是否兴奋？│
└────────┬─────────────┘
         │ 验证通过
┌────────▼─────────────┐
│ 3. 模拟下单           │
│ "如果现在可以购买，     │
│  你愿意付多少钱？"     │
│  ≥30% 给出具体价格    │
└────────┬─────────────┘
         │ 验证通过
┌────────▼─────────────┐
│ 4. 预注册/意向金      │
│ 收集邮箱/微信，        │
│  ≥20人表示"上线后通知我"│
└────────┬─────────────┘
         │ 验证通过
┌────────▼─────────────┐
│ ✅ 进入MVP开发         │
└──────────────────────┘
```

**虚拟MVP不通过的处置**：
- 问题访谈未通过 → 放弃该赛道
- 模拟下单未通过 → 调整定价/方案，重新验证一次；仍不通过则放弃
- 所有候选赛道均未通过 → 重新筛选

## 六、Agent B 监控面板 — 三项消费者硬指标

赛道启动后，Agent B 必须持续监控以下三项指标，纳入自动化周报：

| 指标 | 定义 | 健康阈值 | 预警阈值 | 测量方式 |
|------|------|---------|---------|---------|
| **WAU**（周活跃用户数） | 每周至少使用一次核心功能的独立用户 | 环比 ≥ 0% | 连续2周下降 | 产品埋点自动统计 |
| **NPS**（净推荐值） | "你有多大可能推荐给朋友？"（0-10），Promoters% - Detractors% | ≥ 30 | < 10 | 自动化邮件/应用内调研，每月一次 |
| **流失原因自动归因** | 取消订阅时自动分类：价格/没用/太复杂/找到替代/其他 | 月度流失率 < 5% | 月度流失率 > 8% | 取消流程中的必选问卷 + AI文本分类 |

**Agent B 职责**：
- NPS 低于预警阈值 → 自动触发用户回访（AI邮件/问卷），分析原因，提出改进方案
- 流失率超标 → 自动归因分析，输出 Top 3 流失原因 + 建议对策
- WAU 连续下降 → 检查产品是否出bug、竞品是否异动、用户需求是否变化

## 七、评分计算

```
总分 = Σ(维度得分 × 维度权重)

维度得分 = Σ(子项得分) / 子项数量
```

### 分级标准

| 总分区间 | 评级 | 行动 |
|---------|------|------|
| ≥ 4.0 | S级 | 优先推进，通过虚拟MVP后立即启动开发 |
| 3.5 - 3.9 | A级 | 进入候选池，虚拟MVP验证后决定 |
| 2.5 - 3.4 | B级 | 观望，等待技术或市场条件成熟 |
| < 2.5 | C级 | 放弃 |

## 九、使用说明

1. **每个赛道**独立评估，不横向比较打分（避免锚定效应）
2. **红线检查**和**禁止清单**在打分前进行，通过的才进入打分
3. **虚拟MVP验证**在资源投入前强制执行，无一例外
4. **竞争格局**为定性补充，纳入最终决策建议但不纳入数字评分
5. 框架本身每季度可微调一次（权重、红线），重大调整需用户审批
6. 用户授权在禁止清单之外自由探索赛道，无需每步请示，但周报需说明探索结论
